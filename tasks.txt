
##########################################################################################################################
Objetivo: Convertir el proyecto actual en un Producto Mínimo Viable con sistema de suscripciones.

Arquitectura Actual (basado en el repositorio)

Backend: tauri-api/ → FastAPI + SQLite (contextia.db)
Frontend: tauri-traductor-frontend/ → Vanilla JS + Vite
Base de datos: SQLite (contextia.db) ubicada en la raíz de tauri-api
Estructura backend: carpetas routers, models, middleware, db
Comunicación: El frontend llama al backend vía HTTP (actualmente sin autenticación)

Importante: Todos los cambios deben hacerse respetando la estructura actual de carpetas. No se debe romper el funcionamiento actual de la traducción ni del análisis lingüístico.

Objetivos del MVP

Implementar sistema de usuarios y autenticación
Controlar límites de uso según plan (Free vs Pro)
Implementar sistema de suscripciones
Agregar Historial de traducciones
Crear sección "Mi Cuenta"
Preparar el flujo de onboarding y suscripción

Requerimientos No Funcionales

No romper el flujo actual de traducción/análisis/voz
Mantener el diseño oscuro actual
Mantener responsive (mobile first)
El análisis lingüístico debe estar atado a la traducción (no es una función separada)

##########################################################################################################################
Autenticación y Control de Límites

Requerimientos Funcionales

Los usuarios deben poder registrarse e iniciar sesión.
Debe existir un plan Free por defecto para usuarios no registrados y registrados sin suscripción.
El sistema debe contar las traducciones + análisis lingüístico como una sola unidad.
Los límites se deben aplicar por día (reset a las 00:00 UTC).

Estructura de Base de Datos (Nuevas Tablas)
Deberás crear las siguientes tablas en contextia.db:

users (id, email, password_hash, created_at, is_active)
subscriptions (id, user_id, plan_type, status, current_period_start, current_period_end, stripe_subscription_id)
usage_logs (id, user_id, action_type, created_at, metadata)

Cambios en el Backend (tauri-api/)
Middleware a crear:

middleware/auth.py → JWT authentication
middleware/usage.py → Control de límites por usuario

Routers a modificar/crear:

routers/auth.py → registro, login, logout
routers/usage.py → endpoint para consultar uso restante
Modificar routers/translate.py y routers/analyze.py para que usen el middleware de límites.

Cambios importantes:

Todas las llamadas a traducción y análisis deben pasar por el middleware de uso.
Si el usuario llegó al límite, debe devolver un error claro: {"error": "daily_limit_reached", "remaining": 0}

Cambios en el Frontend

Agregar pantalla de Login / Registro
Guardar JWT en localStorage
Enviar token en el header Authorization: Bearer {token} en todas las llamadas
Mostrar contador de traducciones restantes en la interfaz principal

Sistema de Historial de Traducciones

Requerimientos Funcionales

Guardar cada traducción realizada por usuarios registrados.
El análisis lingüístico debe generarse en el mismo momento de la traducción y guardarse (no on-demand).
El usuario Free solo podrá ver su historial si se registra.
El usuario Pro podrá ver todo su historial completo.

Nueva Tabla en Base de Datos
Crear tabla translation_history:
- id
- user_id (foreign key)
- original_text (text)
- translated_text (text)
- source_language (varchar)
- target_language (varchar)
- audio_base64 (text)        -- guardar el audio generado
- analysis (text)            -- guardar el JSON completo del análisis
- created_at (datetime)

Cambios Técnicos Necesarios
En el Backend (tauri-api/):

Crear routers/history.py con endpoints:
GET /history → obtener historial paginado
GET /history/{history_id} → obtener traducción específica con análisis
POST /translate → modificar para que guarde automáticamente en historial

Cambio crítico: El endpoint de traducción debe generar al mismo tiempo:
La traducción
El audio (speech)
El análisis lingüístico
Y luego guardar todo junto en translation_history.


En el Frontend (tauri-traductor-frontend/):

Agregar ícono de historial en la barra superior (preferiblemente a la derecha).
Crear una nueva página o sección modal/panel para mostrar el historial.
Al hacer clic en un item del historial, mostrar el análisis completo guardado.

Consideraciones Importantes

No cargar todo el historial de una sola vez (usar paginación).
Guardar el analysis como JSON en la base de datos para mantener el formato actual.
El audio_base64 puede ser pesado → evaluar si guardar solo referencia en el futuro.

Mi Cuenta y Flujo de Suscripción

Página de "Mi Cuenta"
Ubicación recomendada: Accesible desde un botón de perfil en la barra superior (junto al ícono de historial).
Contenido que debe mostrar:

Nombre y email del usuario
Plan actual (Free o Pro) con un badge visible
Fecha de vencimiento del plan (si es Pro)
Cantidad de traducciones usadas este mes
Botón "Gestionar Suscripción" (leva a Stripe)
Botón "Ver Historial Completo"
Botón "Cerrar Sesión"

Flujo de Suscripción y Página de Precios
Flujo recomendado:

Usuario se registra o inicia sesión.
Inmediatamente después del login, mostrar un modal ofreciendo suscribirse al plan Pro.
El modal debe tener botón "Suscribirme ahora" y botón "Ahora no".
Siempre debe existir un botón visible "Hacerse Pro" en la barra superior (visible solo para usuarios Free).

Página de Precios (/pricing):
Debe mostrar claramente:

Plan Free (con sus limitaciones)
Plan Pro con los beneficios:
Traducciones ilimitadas
Análisis lingüístico ilimitado
Ejercicios ilimitados
Historial completo
Acceso anticipado a nuevas funciones

Dos botones de suscripción: "Mensual - 9.99 USD" y "Anual - 79.99 USD" (recomendado el anual)

Cambios Técnicos
Backend:

Crear routers/subscription.py para manejar estado de suscripción.
Endpoint para verificar estado actual de suscripción del usuario.

Frontend:

Crear carpeta pages/ o sección para Pricing.
Crear componente de perfil (botón + menú desplegable).
Implementar modal de bienvenida post-registro.


##########################################################################################################################
Modelo de Suscripciones y Plan de Negocio

Estructura Definitiva de Planes
Plan Free

15 traducciones por día
15 análisis lingüísticos por día (atados a las traducciones)
Sin acceso al sistema de ejercicios y práctica
Sin acceso al historial
Uso sin necesidad de registrarse

Plan Pro

Traducciones ilimitadas
Análisis lingüístico ilimitado
Sistema de ejercicios y práctica ilimitada
Historial completo de traducciones
Acceso anticipado a nuevas funciones

Precios Recomendados

Pro Mensual: 9.99 USD por mes
Pro Anual: 79.99 USD por año (equivalente a 6.66 USD/mes)

Recomendación fuerte: Solo ofrecer estos dos planes. No crear un plan intermedio.

Estrategia de Conversión

Después del registro, mostrar modal ofreciendo suscribirse.
Recordatorios suaves cuando el usuario Free se acerca al límite diario.
Enfatizar en el marketing: “Pasa de 15 traducciones por día a uso ilimitado”.

Métricas importantes a medir

Tasa de conversión de Free → Pro
Uso promedio diario de usuarios Free vs Pro
Porcentaje de usuarios que eligen plan anual vs mensual
Retención mes a mes
